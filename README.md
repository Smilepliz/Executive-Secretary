# Руководство пользователя

---

**Открыть приложение:** [https://Smilepliz.github.io/Executive-Secretary/](https://Smilepliz.github.io/Executive-Secretary/)

Одностраничное приложение — демонстрационный прототип редакционного процесса статьи. Данные хранятся в браузере (localStorage).

Ваша задача как пользователя: двигать статью по этапам, выполнять доступные действия и отслеживать историю переходов.

## Что показывает интерфейс

- **Сценарии демонстрации** — быстрый выбор стартового состояния статьи.
- **Этапы процесса** — визуальная шкала этапов: антиплагиат, предоставление документов, рецензенты, рецензирование, публикация.
- **Текущий контекст** — текущий этап и текущий статус статьи.
- **Доступные действия** — только те действия, которые сейчас разрешены.
- **История** — журнал изменений статусов и источников переходов.

## Основная логика работы

- В каждый момент времени статья имеет **один текущий статус**.
- Каждый статус принадлежит определенному этапу процесса.
- Нажимать можно только те действия, которые доступны для текущего статуса.
- После действия статус обновляется, и интерфейс пересчитывает:
  - текущий этап;
  - список доступных действий;
  - историю.

## Как работать в прототипе

1. Выберите нужный сценарий в блоке **Сценарии демонстрации**.
2. Посмотрите блок **Текущий контекст**, чтобы понять, где находится статья.
3. Выполните одно из действий в блоке **Доступные действия**.
4. Проверьте изменения в блоке **История**.
5. Повторяйте, пока не дойдете до нужного результата (публикация, возврат на доработку, отклонение и т.д.).

## Работа с этапами (демо-режим)

- Клик по любому этапу открывает окно выбора статуса внутри выбранного этапа.
- Это ручной переход для проверки разных состояний в интерфейсе.
- После подтверждения новый статус применяется сразу, а в истории появляется запись.

Важно: это демонстрационная возможность, чтобы быстро тестировать ветки процесса.

## Как читать историю

- История показывает переходы от нового к старому (сверху свежие записи).
- По записи видно:
  - какое действие выполнено;
  - из какого статуса в какой был переход;
  - источник перехода.
- Источник `system` означает автоматический переход, не ручной клик пользователя.

## Типовой сценарий прохождения статьи

Обычно путь выглядит так:

1. Ожидает запуска проверки
2. Проверка на антиплагиат идет
3. Антиплагиат пройден
4. **Предоставление документов** — ожидание материалов от автора
5. Материалы получены, готово к назначению рецензентов
6. Подбор и назначение рецензентов
7. Рецензирование
8. Один из исходов:
   - запрос правок автору;
   - принятие статьи;
   - отклонение статьи
9. Подготовка к публикации
10. Публикация

## Что делать, если кнопок действий нет

Если в блоке **Доступные действия** нет кнопок:

- проверьте текущий статус;
- переключите сценарий демонстрации;
- используйте клик по этапу для ручного перехода в другой статус (демо-режим).

### Отправка запроса рецензенту

**Самый простой вариант (без регистраций):** кнопка **«Отправить письмо»** кладёт в буфер адрес, тему и текст — откройте почту в браузере и вставьте (**Ctrl+V**).

Из браузера **нельзя** самостоятельно подключиться к SMTP — это ограничение безопасности. Чтобы письмо уходило **с сайта без вставки вручную**, можно подключить [EmailJS](https://www.emailjs.com/) (нужна регистрация и секреты в GitHub).

1. Зарегистрируйтесь в EmailJS, создайте **Email Service** и **Email Template**.
2. В шаблоне используйте переменные: `{{to_email}}` (получатель), `{{subject}}`, `{{message}}` (в коде для проверки передаётся текст «тест»), `{{reviewer_name}}`, `{{article_title}}`.
3. Скопируйте `.env.example` в `.env.local` и укажите `VITE_EMAILJS_PUBLIC_KEY`, `VITE_EMAILJS_SERVICE_ID`, `VITE_EMAILJS_TEMPLATE_ID`.
4. Пересоберите приложение (`npm run build`).

#### GitHub Pages и переменные EmailJS

Чтобы сборка на GitHub «видела» ключи, их задают как **секреты Actions** (они подставляются в шаг `npm run build` в `.github/workflows/deploy-pages.yml`).

**Где это в интерфейсе GitHub (англ. меню):**

1. Откройте **сам репозиторий** (не профиль и не организация сверху).
2. Вкладка **Settings** (настройки **репозитория** — её видят только владелец или пользователь с правами администратора).
3. Слева в блоке **Security**: **Secrets and variables** → **Actions**.
4. **New repository secret** — по одному секрету на каждое имя: `VITE_EMAILJS_PUBLIC_KEY`, `VITE_EMAILJS_SERVICE_ID`, `VITE_EMAILJS_TEMPLATE_ID`.

**Если пункта Secrets / вкладки Settings у репозитория нет:**

- вы смотрите не тот репозиторий или не тот аккаунт;
- у вас нет прав **администратора** в этом репозитории (тогда секреты может добавить только владелец);
- у форка или у репозитория организации действуют ограничения — уточните у владельца.

**Без доступа к Secrets** на сайте GitHub Pages EmailJS в автосборке не подставится — останется режим **mailto**. Локально всё равно можно проверить EmailJS: файл `.env.local` (см. `.env.example`) и `npm run dev` / `npm run build` на своём компьютере.
